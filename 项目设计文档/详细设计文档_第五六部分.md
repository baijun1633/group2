### 2、类的详细设计描述

#### 2.1 AuthController

**类名**: `AuthController`  
**路径**: `/api/v1/auth`  
**职责**: 处理用户认证相关请求，包括注册、登录和Token刷新

| 方法名 | HTTP方法 | 路径 | 参数 | 返回值 | 说明 |
|--------|----------|------|------|--------|------|
| register | POST | /register | RegisterRequest | ApiResponse\<AuthResponse\> | 用户注册，校验用户名唯一性，密码加密存储，返回用户信息和Token |
| login | POST | /login | LoginRequest | ApiResponse\<AuthResponse\> | 用户登录，验证用户名密码，返回accessToken和refreshToken |
| refreshToken | POST | /refresh | RefreshRequest | ApiResponse\<AuthResponse\> | 刷新Token，验证refreshToken有效性，返回新Token对 |

**详细描述**:
AuthController是系统认证的核心控制器，负责处理用户的注册、登录和Token刷新请求。通过依赖注入AuthService来完成业务逻辑处理。所有接口均采用POST方法，请求体使用JSON格式，返回统一的ApiResponse响应结构。

---

#### 2.2 BooksController

**类名**: `BooksController`  
**路径**: `/api/v1/books`  
**职责**: 处理图书相关的公开查询请求

| 方法名 | HTTP方法 | 路径 | 参数 | 返回值 | 说明 |
|--------|----------|------|------|--------|------|
| searchBooks | GET | /search | keyword, page, size | ApiResponse\<PageResponse\<Books\>\> | 多维度搜索，支持按书名、作者、ISBN模糊匹配 |
| filterBooks | GET | / | categoryId, tag, sortBy, order, page, size | ApiResponse\<PageResponse\<Books\>\> | 分类/标签筛选+排序 |
| getBookDetail | GET | /{bookId} | bookId | ApiResponse\<Map\> | 获取图书详情，包含评分和购书链接 |

**详细描述**:
BooksController提供图书的公开查询接口，支持多维度搜索、分类筛选和详情查看。搜索接口支持按书名、作者、ISBN进行模糊匹配，返回分页结果。筛选接口支持按分类、标签筛选，并可按热度、评分、出版日期排序。

---

#### 2.3 UsersController

**类名**: `UsersController`  
**路径**: `/api/v1/users`  
**职责**: 处理用户个人信息相关请求

| 方法名 | HTTP方法 | 路径 | 参数 | 返回值 | 说明 |
|--------|----------|------|------|--------|------|
| getCurrentUser | GET | /me | - | ApiResponse\<Map\> | 获取当前登录用户信息，包含偏好标签 |
| updatePreferences | PUT | /me/preferences | Map\<String, List\<String\>\> | ApiResponse\<String\> | 更新用户偏好标签 |

**详细描述**:
UsersController处理用户个人信息的查询和更新。getCurrentUser方法从SecurityContext获取当前用户ID，查询用户完整信息并解析JSON格式的偏好标签。updatePreferences方法接收偏好标签列表，序列化为JSON后更新到数据库。

---

#### 2.4 RatingController

**类名**: `RatingController`  
**路径**: `/api/v1/books/{bookId}/rating`  
**职责**: 处理图书评分相关请求

| 方法名 | HTTP方法 | 路径 | 参数 | 返回值 | 说明 |
|--------|----------|------|------|--------|------|
| submitRating | POST | / | bookId, score | ApiResponse\<String\> | 提交评分（1-5分），支持新增和更新 |
| getRatingStats | GET | / | bookId | ApiResponse\<Map\> | 获取图书评分统计（平均分、评分分布） |

**详细描述**:
RatingController负责图书评分的提交和查询。submitRating方法实现upsert逻辑，首次评分创建记录，再次评分更新记录，同时自动更新books表的avg_rating和rating_count字段。getRatingStats方法返回图书的平均评分和各分数段的评分分布。

#### 2.5 AdminBooksController

**类名**: `AdminBooksController`  
**路径**: `/api/v1/admin/books`  
**职责**: 管理员图书管理接口

| 方法名 | HTTP方法 | 路径 | 参数 | 返回值 | 说明 |
|--------|----------|------|------|--------|------|
| createBook | POST | / | Books | ApiResponse\<Books\> | 新增图书 |
| updateBook | PUT | /{bookId} | bookId, Books | ApiResponse\<Books\> | 编辑图书信息 |
| deleteBook | DELETE | /{bookId} | bookId | ApiResponse\<String\> | 删除图书 |
| batchImport | POST | /batch | MultipartFile | ApiResponse\<Map\> | 批量导入图书（CSV格式） |

**详细描述**:
AdminBooksController提供管理员专属的图书管理接口，包括图书的增删改和批量导入功能。所有接口需要ADMIN角色权限。batchImport方法支持CSV文件解析，返回导入成功/失败统计。

#### 2.6 KgController

**类名**: `KgController`  
**路径**: `/api/v1/kg`  
**职责**: 知识图谱查询和可视化接口

|--------|----------|------|------|--------|------|
| queryGraph | GET | /graph/{entityId} | entityId, depth | ApiResponse\<Map\> | 获取图谱可视化数据（nodes和edges） |

**详细描述**:

### 2、类的详细设计描述

#### 2.1 AuthServiceImpl

**类名**: `AuthServiceImpl`  
**实现接口**: `AuthService`  
**职责**: 处理用户认证业务逻辑

| 方法名 | 参数 | 返回值 | 说明 |
|--------|------|--------|------|
| register | RegisterRequest | AuthResponse | 用户注册：校验密码确认、检查用户名唯一性、密码BCrypt加密、生成Token |
| login | LoginRequest | AuthResponse | 用户登录：验证用户名密码、检查用户状态、生成Token |
| refreshToken | RefreshRequest | AuthResponse | Token刷新：验证refreshToken有效性、检查Token类型、生成新Token对 |

**详细描述**:
AuthService是系统认证的核心服务实现类，依赖UsersMapper进行用户数据操作，PasswordEncoder进行密码加密，JwtTokenProvider进行Token生成和验证。register方法实现了完整的注册流程，包括参数校验、用户名唯一性检查、密码加密和Token生成。login方法验证用户凭据并返回Token对。refreshToken方法验证refreshToken有效性后生成新的Token对。

## 三、推荐算法设计

### 3.1 协同过滤推荐（ItemCF）

协同过滤推荐基于"物以类聚，人以群分"的理念，其核心思想是依据用户之间的相似性来推荐物品。在本算法中，通过分析用户的阅读行为（评分、书架收藏），找出与目标用户具有相似阅读口味的其他用户群体。具体而言，若用户A和用户B在阅读的书籍类型、评分模式等方面存在较高的重合度，那么就认为这两个用户具有相似性。基于这种相似性，从与目标用户相似的用户的高分图书中，筛选出目标用户尚未阅读的图书进行推荐，以此帮助目标用户发现可能感兴趣的书籍，实现个性化的图书推荐。

#### （一）实现步骤

**第一步：构建用户-图书评分矩阵**

获取系统中所有用户的评分记录和书架数据，构建用户-图书评分矩阵。评分来源包括：
1. 用户的显式评分（book_ratings表，1-5分）
2. 用户的书架行为（想读=1分、在读=2分、已读=3分）

**第二步：计算用户相似度**

针对每一个其他用户，精确计算其与目标用户的相似度。采用余弦相似度计算公式：

```
similarity(A, B) = (Σ r_Ai × r_Bi) / (√Σ r_Ai² × √Σ r_Bi²)
```

其中 r_Ai 和 r_Bi 分别表示用户A和用户B对图书i的评分。

**第三步：选取相似用户**

完成对所有其他用户与目标用户的相似度计算后，将相似度大于0的用户按照相似度从高到低进行排序，从中选取前K个用户（默认K=10），作为与目标用户相似的用户群体。

**第四步：生成推荐**

从相似用户的高分图书中为目标用户推荐图书：
1. 获取目标用户已经阅读过的图书ID集合，以此作为过滤条件
2. 遍历每个相似用户的高分图书（评分≥4分）
3. 过滤掉目标用户已阅读的图书
4. 按预测评分加权排序
5. 每个相似用户最多提供5个推荐图书
6. 最终推荐列表数量不超过设定上限（默认20本）

### 3.2 基于路径的知识图谱推荐

本算法基于知识图谱构建，依托Neo4j图数据库存储和管理数据，旨在为用户推荐合适的图书。知识图谱借助节点（涵盖图书、作者、出版社、分类、标签等）和关系（如图书由作者编写、图书属于某分类、图书被打上某标签等）呈现数据关联。

算法运行时，依据用户的阅读历史、评分记录和偏好标签，展开图书的匹配推荐。其核心在于计算各图书与用户兴趣的匹配分数，该分数综合考量多方面因素：

#### （一）匹配分数计算

**1. 标签匹配权重**

用户偏好标签与图书标签的重合度。若用户偏好标签与图书标签存在交集，权重设为1.2；无交集则为1.0，以此体现标签适配性差异。

```
tagWeight = 1.2 (有交集) / 1.0 (无交集)
```

**2. 分类匹配权重**

用户阅读历史中高频分类与图书分类的匹配程度。若图书分类属于用户高频分类Top3，权重为1.5；属于Top5，权重为1.2；其他为1.0。

```
categoryWeight = 1.5 (Top3) / 1.2 (Top5) / 1.0 (其他)
```

**3. 作者匹配权重**

用户阅读历史中高频作者与图书作者的匹配程度。若图书作者属于用户高频作者，权重为1.3；否则为1.0。

```
authorWeight = 1.3 (匹配) / 1.0 (不匹配)
```

**4. 评分权重**

图书的平均评分和评分人数综合权重。评分越高、评分人数越多，权重越大。

```
ratingWeight = (avgRating / 5.0) × log(ratingCount + 1)
```

**5. 关系路径权重**

基于知识图谱的多跳路径推理，路径越短权重越大。

```
pathWeight = 1.0 / hops
```

#### （二）多跳推理路径

**3跳路径示例**:

```
路径1: 用户 →[喜欢]→ Tag "科幻" →[tagged_as]→ Book "银河帝国" →[written_by]→ Author "阿西莫夫" →[written_by]→ Book "基地"
路径2: 用户 →[评分高]→ Book "三体" →[belongs_to]→ Category "科幻小说" →[belongs_to]→ Book "流浪地球" →[tagged_as]→ Tag "刘慈欣"
路径3: 用户 →[喜欢]→ Author "刘慈欣" →[written_by]→ Book "球状闪电" →[related_to]→ Book "超新星纪元"
```

#### （三）综合匹配分数计算

通过将上述权重相乘，得出每本图书与用户兴趣的匹配分数：

```
matchScore = tagWeight × categoryWeight × authorWeight × ratingWeight × pathWeight
```

随后，算法依此分数对查询结果从高到低排序，把图书信息和匹配分数进行封装，最终反馈给用户，实现精准的图书推荐服务。

#### （四）推荐理由生成

基于推理路径生成自然语言推荐理由，包含推理路径记录（节点ID列表）：

| 推荐理由模板 | 示例 |
|-------------|------|
| 因为您喜欢《{book1}》，同属{category}题材的《{book2}}》也值得一读 | 因为您喜欢《三体》，同属科幻题材的《银河帝国》也值得一读 |

| 与您阅读过的《{book1}》相关的《{book2}》推荐给您 | 与您阅读过的《三体》相关的《流浪地球》推荐给您 |

## 二、详细表结构

### 1、users 表结构（用户表）

| 字段名 | 类型 | 长度 | 允许空 | 默认值 | 说明 |
|--------|------|------|--------|--------|------|
| user_id | BIGINT | - | 否 | 自增 | 用户ID，主键 |
| username | VARCHAR | 50 | 否 | - | 用户名，唯一 |
| password | VARCHAR | 255 | 否 | - | 密码（BCrypt加密） |
| nickname | VARCHAR | 50 | 是 | - | 昵称 |
| email | VARCHAR | 100 | 是 | - | 邮箱 |
| phone | VARCHAR | 20 | 是 | - | 手机号 |
| avatar | VARCHAR | 255 | 是 | - | 头像URL |
| role | VARCHAR | 20 | 否 | 'USER' | 角色：ADMIN-管理员，USER-普通用户 |
| preference_tags | JSON | - | 是 | NULL | 偏好标签，如 ["科幻","悬疑"] |
| status | TINYINT | - | 否 | 1 | 状态：0-禁用，1-启用 |
| create_time | DATETIME | - | 否 | CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | - | 否 | CURRENT_TIMESTAMP | 更新时间 |

**索引**:
- PRIMARY KEY (user_id)
- UNIQUE KEY uk_username (username)

### 3、Neo4j 图数据库关系说明

本系统除MySQL关系型数据库外，还使用Neo4j图数据库存储知识图谱数据。Neo4j中的节点和关系不使用外键约束，而是通过Cypher查询语言进行关联查询。

**Neo4j 节点类型与对应MySQL表**:

| Neo4j节点类型 | 对应MySQL表 | 关键属性 |
|--------------|-------------|----------|
| Book | books | bookId, title, isbn |
| Author | (从books.author字段提取) | name |
| Publisher | (从books.publisher字段提取) | name |
| Category | categories | categoryId, name |
| Tag | (从books.tags JSON字段提取) | name |

**Neo4j 关系类型**:

| 关系类型 | 起始节点 | 目标节点 | 说明 |
|----------|----------|----------|------|
| written_by | Book | Author | 图书由作者编写 |
| belongs_to | Book | Category | 图书属于某分类 |
| published_by | Book | Publisher | 图书由某出版社出版 |
| tagged_as | Book | Tag | 图书被打上某标签 |
| related_to | Book | Book | 图书之间存在关联 |
